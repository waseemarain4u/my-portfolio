5 Flutter performance mistakes I made so you don't have to

Jank, memory leaks, unnecessary rebuilds — a tour of the performance issues I've debugged over the years and how to avoid them.

WA

Waseem Akram

Senior Flutter Developer & Lead Developer

Flutter apps are fast by default — until they aren't. I've debugged enough jank, memory pressure issues, and frame drops across production apps to have strong opinions about where Flutter performance problems actually come from.

1. Rebuilding the entire widget tree on state changes

This is the most common mistake I see in Flutter codebases. A single ChangeNotifier or BLoC state update triggers a rebuild at the top of the tree, and Flutter re-renders widgets that have nothing to do with the changed data. The fix is straightforward: push state as low as possible, use const constructors everywhere you can, and wrap expensive subtrees in RepaintBoundary.

// Don't do this — rebuilds the whole screen
Consumer<AppState>(builder: (ctx, state, _) => Scaffold(...));

// Do this — rebuilds only the widget that needs the data
Consumer<AppState>(builder: (ctx, state, _) => Text(state.username));

2. Heavy work on the main isolate

JSON parsing, image processing, and database queries run on the main thread by default. On large payloads this causes visible frame drops. Move anything that takes more than a few milliseconds to a separate isolate using compute() or Flutter's Isolate API.

3. Not disposing controllers and streams

TextEditingControllers, AnimationControllers, StreamSubscriptions — all of these leak memory if you don't dispose them. I've seen apps with gradual memory growth that was entirely caused by controllers created in initState and never cleaned up. Always override dispose(). Always.

4. Loading full-resolution images

Loading a 4K photo to display in a 50x50 avatar thumbnail is a surprisingly common performance issue. Use cacheWidth and cacheHeight parameters on Image.network to decode images at the size they'll actually be displayed. On image-heavy screens, this alone can cut memory usage significantly.

5. Animating with setState

Using setState inside an animation callback rebuilds the whole widget on every frame. Use AnimatedBuilder or AnimatedWidget instead — they limit the rebuild to only the animated portion of the tree. On complex screens this is the difference between smooth 60fps and visible jank.

The Flutter DevTools performance tab is your best friend for diagnosing all of these. Profile in release mode — debug mode is much slower and will give you misleading results.

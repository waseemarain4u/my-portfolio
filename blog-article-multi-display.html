Multi-Display Magic in Android: Sending Messages to a Second Screen

Android's multi-display API is underused and underrated. Here's how to launch activities on a secondary display and communicate between them.

WA

Waseem Akram

Senior Flutter Developer & Lead Developer

Most Android developers never think about secondary displays. But if you've ever worked on retail kiosks, digital signage, presentation apps, or Android Automotive, you've already needed this. Android's multi-display support has been production-ready since API 26, and the API is simpler than most people expect.

When does this matter?

The use cases are more common than they appear. A point-of-sale system where the cashier sees an order management screen and the customer faces a payment/receipt display. An Android TV app driving both a main screen and a status bar. A vehicle infotainment system where the driver and passenger see different content. All of these require launching and controlling activities on a display that isn't the primary one.

Detecting available displays

The entry point is DisplayManager, which you get from the system service. It gives you a list of all connected displays and lets you register a listener for display connect/disconnect events.

val displayManager = getSystemService(Context.DISPLAY_SERVICE) as DisplayManager
val displays = displayManager.displays

// Find secondary displays
val secondaryDisplays = displays.filter { it.displayId != Display.DEFAULT_DISPLAY }

Always listen for display changes at runtime — external displays can connect and disconnect at any time, and your app needs to handle both gracefully.

Launching an Activity on a specific display

Once you have the target display's ID, you create a ActivityOptions bundle specifying the launch display, then pass it to startActivity.

val options = ActivityOptions.makeBasic().apply {
    launchDisplayId = targetDisplay.displayId
}
val intent = Intent(this, SecondScreenActivity::class.java)
startActivity(intent, options.toBundle())

That's the core of it. The activity launches on the specified display as a normal Android activity, with its own lifecycle, window, and view hierarchy.

Communicating between screens

This is where it gets interesting. Since both activities run in the same process, you have several clean options for communication:

  • Shared ViewModel: Both activities share a ViewModel scoped to the application — state changes in one are immediately reflected in the other via LiveData or StateFlow.
  • Local Broadcast: Use LocalBroadcastManager to send events between activities without any cross-process overhead.
  • EventBus / shared repository: A singleton repository with a shared StateFlow is the cleanest architecture-friendly approach.

The shared ViewModel approach integrates most naturally with modern Android architecture. Both the primary and secondary screen activities observe the same state, and any update from either side propagates automatically.

Handling display disconnection

A secondary display can disappear at any time — the user unplugs HDMI, a Bluetooth display times out, or the device switches modes. Register a DisplayManager.DisplayListener and handle onDisplayRemoved to gracefully wind down any activities running on the disconnected display. Failing to do this results in orphaned activities and potential ANRs.

Multi-display development opens up a class of app experiences that most Android developers haven't explored. The API is well-documented and surprisingly approachable — the hard part is designing the UX, not writing the code.

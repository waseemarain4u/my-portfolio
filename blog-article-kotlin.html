Kotlin vs Java in 2025: when to still choose Java

Kotlin is the default, but there are still real-world situations where Java makes more sense for your Android project.

WA

Waseem Akram

Senior Flutter Developer & Lead Developer

Every Android developer today defaults to Kotlin — and for most projects, that's the right call. But after 10+ years of Android development, I've seen enough codebases to know that Java still earns its place in specific situations.

Why Kotlin won

Kotlin's coroutines, null safety, extension functions, and concise syntax make it genuinely better for most modern Android work. Google endorsed it in 2017, and by 2019 it became the preferred language for Android. If you're starting a new project today, Kotlin is the obvious choice.

When Java still makes sense

Here are the real scenarios where I've reached for Java in recent years:

  • Existing Java codebases: Migrating a 100k-line Java codebase to Kotlin incrementally is possible, but risky without a clear plan. Sometimes maintaining Java consistency is more pragmatic than a partial migration.
  • Library development: If you're writing a library that needs to be consumed by both Java and Kotlin callers, pure Java avoids the interop friction entirely.
  • Team familiarity: A team that deeply knows Java and is new to Android will ship faster in Java — at least initially. Architecture matters more than language in the short term.
  • Performance-critical native code: For JNI-heavy components, Java's simpler memory model can sometimes be easier to reason about.

The interop reality

Kotlin and Java interop is excellent — you can call Java from Kotlin and vice versa in the same module. But interop isn't free. Nullability annotations, SAM conversions, and property access syntax all need attention. In a pure-Java module consumed by Kotlin, you'll want to annotate everything with @NonNull and @Nullable to get proper null safety on the Kotlin side.

My actual recommendation

New project? Kotlin, no question. Existing Java project? Migrate new features to Kotlin, leave stable old code alone. Mixed team with Java seniors? Start with Kotlin for new modules and invest in upskilling — the productivity gains compound quickly once the team is comfortable.

The language debate matters less than most developers think. Clean architecture, good testing habits, and maintainable code are language-agnostic.
